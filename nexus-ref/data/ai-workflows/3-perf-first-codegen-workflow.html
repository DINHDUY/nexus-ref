Module 1 — What & Why · 3 min
Performance-First Code Generation
A 9-agent workflow that researches, constitutionalizes, and implements high-performance code from a natural language request — with correctness guaranteed by TDD and performance guaranteed by iterative optimization loops.
Core Idea
Research Before Code
A performance "constitution" of enforceable rules is derived from algorithm research before any implementation starts — not retrofitted after slow code is caught.
Good Fit
When to Use
Parsers, encoders, algorithms, library development with quantified throughput/latency/memory goals. Any code that will be benchmarked or profiled.
Not For
When to Skip
Quick prototypes, UI components, standard business logic, or one-off scripts where performance is "nice to have" but not measured.

Module 3 — The Agents · 3 min
9 Specialized Agents
Each agent owns exactly one stage. Sonnet agents handle complex reasoning and code generation; Fast agents handle structured, rule-following coordination tasks. Two agents are read-only.
Agent Model R/O Primary Role
perf.orchestrator sonnet ✗ Coordinates all 8 subagents through the 7-stage pipeline; manages Loop B
perf.researcher sonnet ✓ Researches optimal algorithms, data structures, and language-specific optimizations
perf.constitution-writer fast ✗ Converts research into 8-section enforceable coding rules and anti-pattern bans
perf.spec-writer sonnet ✗ Authors formal spec + 3 test files: correctness, performance benchmarks, property-based
perf.planner fast ✗ Generates TDD implementation plan with build order and test-to-step mappings
perf.task-decomposer fast ✗ Breaks plan into atomic tasks: <30 lines, one test, one file, DAG dependencies
perf.loop-controller fast ✗ Inner loop manager: drives implement → test → retry cycles (max 5)
perf.implementer sonnet ✗ Writes code per atomic tasks following constitution rules; fixes failures on retry
perf.test-runner fast ✓ Executes correctness tests + benchmarks; generates structured test-report.json
◈
Atomicity rule: Every implementer task satisfies exactly one test, modifies at most one source file, and stays under 30 lines of code.

Module 4 — In Practice · 3 min
Invoking the Workflow
Always address @perf.orchestrator. A strong request includes: what to build, language + version, performance goal with units, and expected input scale.
Request template
@perf.orchestrator [what to build] in [language + version].
Target [throughput / latency / memory goal with units].
Input scale: [file sizes or record counts]. Context: [use-case note].
Example 1 — CSV Parser (Python)
@perf.orchestrator Generate a CSV parser in Python 3.12
that processes 100 MB/s for typical data files.
Support RFC 4180 quoting and handle edge cases
like empty fields and escaped quotes.
Example 2 — Matrix Multiplication (Rust)
@perf.orchestrator Implement matrix multiplication in Rust 1.75
for f64 matrices. Target 50 GFLOPS single-threaded.
Memory usage must not exceed 2x input size.
Example 3 — JSON Parser (Go)
@perf.orchestrator Create a zero-allocation JSON parser
in Go 1.21 for real-time systems.
Parse latency must be under 5ms p99 for documents up to 100KB.
⚠
Execution time: Small (100–300 lines): 10–20 min · Medium (500–1000 lines): 30–60 min · Large (1000+ lines): 1–3 hrs. Each Loop B iteration adds 50–75% to base time.

Module 5 — Mastery · 3 min
Best Practices & Knowing the Limits
What separates a great perf workflow invocation from a frustrating one — and what to do when loops plateau, fail, or reach their limits.
①
Quantify everything
Always include units: MB/s for throughput, ms p99 for latency, MB for memory. Vague targets produce vague constitutions and vague benchmarks.
②
Specify scale
Mention expected input sizes (KB / MB / GB). This determines which algorithm class the researcher recommends — O(n log n) vs O(n) vs SIMD approaches differ by scale.
③
Provide use-case context
"This will process application log files" focuses research far better than a bare function signature. The researcher adjusts algorithm selection based on real-world patterns.
④
Read Loop B outcomes correctly
Gap <10% + improvement <2% = acceptable plateau. Gap >10% after 3 iterations = algorithmic ceiling — consider manual constitution redesign.
◈
If Loop A keeps failing: Check for contradictory constitution rules, overly strict tests, or tasks exceeding 30 lines. Manual task-graph adjustment may be needed.
Loop B detects a "plateau" when which condition is true?
AAll tests in Loop A fail on the final iteration
BAll performance metrics improved by less than 2% versus the previous Loop B iteration
CThe constitution file contains more than 8 sections
