Permission Evaluation Flow — How Every Tool Call Is Decided
⚙️ The Decision Pipeline (Applied in This Exact Order)
① Deny rules
deny always wins
→
② Permission mode
bypass/acceptEdits/…
→
③ Allow rules
settings.json allow
→
④ canUseTool
runtime prompt user
→
⑤ dontAsk mode
deny if unresolved
StepWhat HappensNotes
① Deny rules Check deny array in settings.json + --disallowedTools CLI flag. If matched → BLOCKED. No further evaluation. Deny always overrides allow at any level. Cannot be overridden by any lower layer.
② Permission mode bypassPermissions → approve all. acceptEdits → approve file ops. Others fall through. Hooks still run even in bypassPermissions and can block.
③ Allow rules Check allow array in settings.json + --allowedTools CLI. If matched → APPROVED. First match wins. Rules evaluated top-to-bottom.
④ canUseTool callback Runtime prompt to user for approval. Session-level "Always allow" adds to allow list. Skipped in dontAsk mode.
⑤ dontAsk fallback If dontAsk mode and no rule matched → DENIED without prompting. TypeScript SDK only. Use for fixed headless agent surfaces.
⚠ Critical Rule: Deny Always Wins If a tool is denied at any settings layer (enterprise, user, project, local), no other layer can allow it. --allowedTools CLI flag cannot override a managed-settings deny. A deny in project settings overrides an allow in user settings.

Rule Syntax — allow · deny · ask
🔧 settings.json Structure & Rule Arrays
Complete settings.json anatomy 
{
  "model": "claude-sonnet-4-6",
  "permissions": {
    "defaultMode": "acceptEdits",
    "allow": [
      "Read",                       // all reads
      "Edit(src/**)",               // glob path
      "Bash(npm run *)",            // bash glob
      "Bash(git commit *)",         // bash exact prefix
      "WebFetch(domain:github.com)", // domain rule
      "mcp__puppeteer",             // entire MCP server
      "Agent(reviewer)"             // named subagent
    ],
    "deny": [
      "Read(.env*)",                 // block .env files
      "Read(secrets/**)",           // block dir tree
      "Edit(//etc/**)",             // absolute path
      "Bash(rm -rf *)",             // dangerous command
      "Bash(sudo *)",               // no sudo
      "Bash(git push *)"            // always ask before push
    ],
    "ask": [
      "Bash(git push *)",           // prompt user first
      "Bash(npm install *)",        // install = ask
      "WebFetch"                    // all fetches = ask
    ],
    "additionalDirectories": [
      "../shared-lib", "../docs"
    ]
  }
}
📐 All Rule Syntaxes by Tool
Bash rules — glob patterns, space boundary matters 
── Exact command ──────────────────────────────── 
"Bash(npm run build)"      # exact match only

── Prefix glob ────────────────────────────────── 
"Bash(npm run *)"          # npm run test, npm run dev…
"Bash(git * main)"         # git checkout main, git merge main
"Bash(* --version)"        # any --version command

── Space-boundary rule (important!) ─────────────
"Bash(ls *)"    # ✓ matches: ls -la    ✗ does NOT match: lsof
"Bash(ls*)"     # ✓ matches: ls -la    ✓ ALSO matches: lsof

── ⚠ DEPRECATED (still works but avoid) ─────────
"Bash(npm:*)"   # old :* suffix syntax → use space+* instead
Read / Edit rules — gitignore-style paths 
"Read"               # all reads (no parens = all)
"Edit(src/**)"       # recursive under src/
"Edit(*.ts)"         # .ts files anywhere (one level)
"Read(.env*)"        # .env, .env.local, .env.prod

── Absolute path: use double leading slash ───────
"Edit(//tmp/scratch.txt)"  # absolute: /tmp/scratch.txt
"Read(//Users/alice/**)"   # absolute dir tree
# "Read(/Users/alice)" → NOT absolute: relative to project root!
WebFetch / MCP / Agent rules 
"WebFetch(domain:docs.anthropic.com)"  # allow domain
"WebFetch"                              # all fetches

"mcp__puppeteer"              # all tools from server
"mcp__puppeteer__*"           # wildcard (same effect)
"mcp__github__create_pr"      # specific tool

"Agent(my-custom-agent)"      # named subagent
⚠ Bash argument patterns are fragile Bash(curl http://github.com/ *) won't match curl -X GET https://… because options may precede the URL. For reliable URL filtering, block Bash(curl *) in deny and use WebFetch(domain:github.com) instead.
Permission Modes — 5 Operating Modes
🎛 5 Modes — Match the Mode to Your Context
default
shift+tab → cycle · /mode default
Prompts for approval on every modification. Read ops are auto-allowed. Safest for unknown codebases or when reviewing AI changes carefully.
acceptEdits
shift+tab · "defaultMode": "acceptEdits"
Auto-approves all file operations (Read, Write, Edit). Bash commands that aren't filesystem ops still require approval. Best for prototyping or trusted isolated dirs.
plan
/plan · --mode=plan · shift+tab
Claude reads and proposes a plan but cannot execute tools or modify files. Use for exploration, architecture discussion, and reviewing intent before execution.
dontAsk
TypeScript SDK only · permission_mode="dontAsk"
Pre-approved tools from allow rules run; everything else is silently DENIED without prompting. Best for headless agents with a fixed, explicit tool surface. Not available in Python SDK.
bypassPermissions
--dangerously-skip-permissions · only in sandboxed env
ALL tool calls approved without any prompts. Hooks still execute and can block. Use ONLY inside Docker/container with no sensitive data. Never in production or unprotected directories.
ℹ Cycle modes in-session Press shift+tab to cycle through modes without leaving the session. Set "defaultMode" in settings.json to persist across sessions. The mode can also be changed dynamically mid-session via SDK streaming.
⚠ bypassPermissions + allowed_tools gotcha allowed_tools=["Read"] + permission_mode="bypassPermissions" still approves ALL tools including Bash/Write/Edit — allowed_tools only pre-approves, it does not restrict. To actually block tools in bypass mode, use disallowed_tools (deny rules).
Hooks — Lifecycle Automation
🔗 All Hook Events — Complete Reference
EventFires WhenMatcher FieldCan Block?
PreToolUse Before any tool call. Can modify input, approve, deny, or inject context. Tool name / pattern e.g. Bash, Edit|Write ✓ exit 2 or permissionDecision: "deny"
PermissionRequest When a permission dialog appears for the user (v2.0.45+). Can auto-approve/deny. Tool name pattern ✓ decision: allow / deny / ask
PostToolUse After successful tool completion. Can provide feedback. Cannot undo the operation. Tool name / pattern ✗ can't undo · can give feedback
PostToolUseFailure When a tool call fails. Useful for structured error logging with full context. Tool name / pattern ✗ informational only
UserPromptSubmit Before Claude processes a user prompt. Can inject additionalContext or block bad prompts. * (all) or text pattern ✓ exit 2 blocks prompt; stdout added as context
Stop When main agent finishes responding. Can force continuation (decision: "block"). — (no matcher) ✓ decision: "block" + reason forces Claude to continue
SubagentStop When a subagent (Task tool) finishes. Subagent Stop hooks in frontmatter auto-convert to this. — (no matcher) ✓ same as Stop
SubagentStart When a subagent spawns. For DB setup, env prep for specific agents. Agent name or * ✗ setup only
SessionStart On new session, resume, /clear, or post-compact. For context injection, env setup. Source field: startup / resume / clear / compact. Session source or * ✗ stdout added as Claude context
SessionEnd On exit, SIGINT, or error. For logging, cleanup, transcript export. — (no matcher) ✗ cleanup only
Notification When Claude sends a notification (permission prompt, idle). Fire desktop alerts. * ✗ informational
PreCompact Before context window compaction. Can inject context to preserve across /compact. — (no matcher) ✗ context injection
Setup Triggered by --init, --init-only, or --maintenance CLI flags. For repo setup tasks. Trigger: init / maintenance ✗ setup only
Hooks Syntax — Config, Exit Codes, JSON Control
⚙️ Hooks Config in settings.json
Full hook entry anatomy 
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",            // tool name / regex
        "hooks": [
          {
            "type": "command",         // command | http | prompt
            "command": "./scripts/validate.sh",
            "timeout": 60             // default: 600s (10min)
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit", // pipe = OR
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\""
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude done!\"'"
          }
        ]
      }
    ]
  }
}
3 hook handler types 
── type: "command" (most common) ────────────────
"type": "command",
"command": "./scripts/validate.sh"
# receives JSON via stdin; exit code controls outcome

── type: "http" (v2.x+) ─────────────────────────
"type": "http",
"url": "http://localhost:8080/hooks/pre-tool",
"timeout": 30,
"headers": { "Authorization": "Bearer $MY_TOKEN" },
"allowedEnvVars": ["MY_TOKEN"]
# POST body = same JSON as stdin for command hooks
# 4xx/5xx = non-blocking error (CANNOT block via status!)
# To block: return 2xx + JSON deny in response body

── type: "prompt" (LLM evaluation) ──────────────
"type": "prompt",
"prompt": "Evaluate if this bash command is safe: $ARGUMENTS"
# Best for Stop/SubagentStop semantic evaluation
🚦 Exit Codes & JSON Control Output
Exit code behaviour 
── exit 0 ────────────────────────────────────────
# Success. stdout parsed as JSON for structured control.
# JSON only processed on exit 0.

── exit 2 ────────────────────────────────────────
# BLOCKING ERROR. stdout ignored. stderr fed to Claude.
# PreToolUse     → BLOCKS the tool call
# UserPromptSubmit → REJECTS the prompt
# Stop           → FORCES Claude to keep working
# Other events   → context-specific blocking

── any other non-zero ────────────────────────────
# Non-blocking error. stderr shown in verbose mode (Ctrl+O).
# NOT fed back to Claude. Execution continues.

⚠ Common pitfall: many CLI tools write errors to stdout,
# NOT stderr. For Claude to receive blocking feedback,
# explicitly redirect:  echo "blocked" >&2; exit 2
JSON structured output (stdout on exit 0) 
── PreToolUse: deny with reason ─────────────────
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",     // allow|deny|ask
    "permissionDecisionReason": "DB writes blocked",
    "updatedInput": { "command": "modified-cmd" },
    "additionalContext": "Extra info for Claude"
  }
}

── Stop: force continuation ─────────────────────
{
  "decision": "block",
  "reason": "Tests still failing. Fix them before stopping."
}

── Stop: allow stop cleanly ─────────────────────
{
  // omit "decision" entirely → Claude stops normally
  "continue": false,             // explicit stop
  "suppressOutput": true          // hide from transcript
}
⚠ HTTP hooks cannot block via status codes A 4xx or 5xx from an HTTP hook logs an error but execution continues. To block, return a 2xx response with permissionDecision: "deny" in the JSON body.

#!/bin/bash  # .claude/hooks/enforce-complete.sh
json_input=$(cat)
# Check if tests pass before allowing stop
if ! pnpm test --passWithNoTests 2>/dev/null; then
  echo '{"decision":"block","reason":"Tests still failing. Fix before stopping."}'
  exit 0   # exit 0 so JSON is parsed

Decision Tree — Where Does This Permission Belong?
🌿 Choose the Right Layer for Your Rule
Is this rule an org-wide security mandate (no one should override)?
├─ YES → Enterprise managed policy file (MDM/plist — sysadmin deploys)
└─ NO  → Is this a one-off CLI session (not persistent)?
   ├─ YES → --allowedTools / --disallowedTools CLI flags (per-invocation)
   └─ NO  → Should it apply to all projects on your machine?
      ├─ YES → ~/.claude/settings.json (user global defaults)
      └─ NO  → Is it project-specific and should be shared with the team?
         ├─ YES → .claude/settings.json (commit to git — team-wide)
         └─ NO  → Is it a personal override for this project only?
            ├─ YES → .claude/settings.local.json (gitignored — your machine only)
            └─ NO  → Need runtime logic or dynamic checks?
                      ├─ YES → Is it before a tool call?
                      │  ├─ YES → PreToolUse hook (can block, modify input, inject context)
                      │  └─ NO  → After a tool call?
                      │        ├─ YES → PostToolUse hook (auto-format, lint, log)
                      │        └─ NO  → Stop / SubagentStop hook (enforce completion)
