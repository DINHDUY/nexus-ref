What Are Cursor Rules?
📐 Rules vs Skills vs Commands

Rules are persistent, always-available instructions that shape every AI interaction in your project. Unlike Skills (on-demand how-to) or Commands (manual prompts), rules are automatically injected into context based on their activation mode.

Rules

  • .mdc (YAML + Markdown)
  • Auto-applied / always-on
  • Standards & style
  • Persistent constraints
  • 4 activation modes
  • Cursor only
  • Glob-scoped

Skills

  • SKILL.md + dirs
  • On-demand (lazy load)
  • How-to knowledge
  • Auto + /slash menu
  • Has scripts/ dir
  • Open standard
  • 20+ agents

Commands

  • .cursor/commands/*.md
  • Manual invoke only
  • Saved prompts
  • Type /name
  • No scripts
  • Cursor only
  • No frontmatter

🏛 Rules Hierarchy — 4 Levels
LevelLocationScope
Global Cursor Settings → Rules for AI All projects, all files. Plain text, not .mdc.
Project .cursor/rules/*.mdc Entire repo. Version-controlled. Recommended.
Subdirectory subdir/.cursor/rules/*.mdc Only active when working inside that folder.
Legacy .cursorrules (root) Deprecated. .mdc files win on conflict. Migrate.
⚠ Conflict Resolution If you have the same rule in both .cursorrules and .cursor/rules/, the .mdc version wins silently. The legacy file is ignored with no error or warning.
ℹ /rules CLI Command Since Jan 8 2026: type /rules in a CLI session to create new rules or edit existing ones directly from the terminal without opening a file.
.MDC File Anatomy
📄 Full .MDC Template
📁 .cursor/rules/my-rule.mdc 
─── YAML FRONTMATTER (required) ─────────────────
---
description: Short description of the rule's purpose
               # shown in Cursor Settings UI
               # used for agent-decision mode matching

globs: src/**/*.ts, src/**/*.tsx
       # comma-separated glob patterns
       # rule auto-attaches when matched file is open
       # ignored if alwaysApply: true

alwaysApply: false
             # true  → always in context, no conditions
             # false → conditional (glob or agent-decides)
             # omitted → still works in agent mode
---
─── MARKDOWN BODY ───────────────────────────────

📋 Frontmatter Fields & Behaviours
FieldTypeBehaviour
description string Shown in Settings UI. Used by agent to decide relevance. Can be empty but reduces auto-detection quality.
globs string Comma-separated glob list. Rule auto-attaches when a matching file is open. Ignored if alwaysApply: true.
alwaysApply boolean true = always in context regardless of file. false or omitted = conditional. Glob filters ignored when true.
① Always
alwaysApply: true
No globs needed. Rule is injected into every single conversation. Use for universal code style standards.
② Auto-attach
globs: src/**/*.ts
alwaysApply: false
Injected automatically when a matching file is open or referenced.
③ Manual
No globs, no alwaysApply.
Attach explicitly with @rule-name in Cmd-K or chat. Good for opt-in standards.
④ Agent Decision
Has description set.
AI reads description and decides whether to apply based on task relevance. Non-deterministic.
⚡ Critical — Silent Failures Missing or malformed YAML frontmatter causes the rule to silently not load — no error, no warning. The closing --- must be present. Validate with npx cursor-lint.

Glob pattern reference 
# Single extension
globs: **/*.ts

# Multiple extensions (comma-separated)
globs: **/*.ts, **/*.tsx, **/*.js

# Scoped to a directory
globs: src/components/**/*.tsx

# Test files only
globs: **/*.test.ts, **/*.spec.ts

# API routes
globs: src/api/**/*.ts, app/api/**/*.ts

# Python files
globs: **/*.py

# Config files
globs: *.config.ts, *.config.js

# Schema / migration files
globs: prisma/**/*.prisma, migrations/**
⚠ Glob behaviour quirk If you set globs: without alwaysApply: true, the rule only attaches when a matched file is actually open. If no file is open, the rule is ignored entirely.

# Code Style

## Principles
- Write self-documenting code
- Prefer readability over cleverness
- One concept per function
- Use early returns to reduce nesting
- DRY but not at the cost of clarity

## Naming
- Variables/functions: camelCase
- Classes/types: PascalCase
- Constants: SCREAMING_SNAKE_CASE
- Files: kebab-case
- Booleans: isX, hasX, canX, shouldX

## Comments
- Comments explain WHY, not WHAT
- No commented-out code in commits
- English only

# TypeScript Standards

## Types
- Never use `any` — use `unknown` instead
- All function params must be typed
- All function returns must be typed
- Prefer `interface` for objects,
  `type` for unions/intersections
- Use branded types for domain IDs:
  type UserId = string & { _brand: 'UserId' }

## Patterns
- Prefer `const` assertions for literals
- Use discriminated unions over optional
- Avoid enums — use const objects instead
- Nullish coalescing over || for defaults

# React Components

## Structure
- Functional components only (no class)
- Props interface named `[Name]Props`
- Default export the component
- Named export for tests

## Hooks
- No logic outside of hooks/handlers
- Custom hooks prefix: use*
- Memoize expensive computations
- useCallback for stable references

## Styling
- Tailwind utility classes only
- No inline styles
- Variant-driven with cva()

## Structure
- AAA: Arrange → Act → Assert

- Describe blocks match the unit name

- Mock at the module boundary
- No test-only code in source files
- No `.only` or `.skip` in commits

## Coverage
- 80% line coverage minimum
- Always test error/edge cases

# Database Rules

## Schema
- All tables need createdAt/updatedAt
- Use UUIDs for all primary keys
- Index all foreign keys
- Never use nullable unless required

## Queries
- Always select only needed fields
- Paginate all list queries
- Never N+1: use include/join
- Wrap mutations in transactions

## Migrations
- Never drop without 2-phase plan
- Always write rollback migration

# Error Handling

## Rules
- Always catch errors explicitly
- Never swallow errors silently
- Use specific error types, not Error
- Error messages: actionable, specific
- Log errors with context (file, fn, input)

## Pattern (Result type)
type Result<T> =
  | { ok: true;  value: T }
  | { ok: false; error: AppError }

## Async
- All async functions return Result<T>
- No unhandled promise rejections
- Handle both network AND logic errors

Global Rules (Settings) vs Project Rules
🌐 Global Rules — Cursor Settings

Global rules are configured in Cursor Settings → Rules for AI. They are plain text (no YAML frontmatter, no .mdc) and apply to every project on your machine.

Example global rules content (plain text) 
<personal_preferences>
- Always use British English spelling
- Prefer functional programming patterns
- Always include type annotations
- Keep responses concise and direct
</personal_preferences>

<code_style>
- Comments in English only
- Follow DRY, KISS, and YAGNI principles
- Write pure functions where possible
- Use strict typing everywhere
</code_style>

<error_handling>
- Always raise errors explicitly
- Use specific error types
- Error messages: clear and actionable
</error_handling>
✦ Global vs Project Keep global rules to personal preferences only (tone, formatting style). All project standards belong in .cursor/rules/ so the whole team benefits and rules are version-controlled.

Decision Tree — Which Activation Mode?
🌿 Mode Selection Guide
Should this rule apply to every single conversation?
├─ YES → Is it a hard security/safety non-negotiable?
│  ├─ YES → alwaysApply: true  (security.mdc, no-secrets.mdc)
│  └─ NO  → Is it a universal coding style or principle?
│     ├─ YES → alwaysApply: true  (code-style.mdc, naming.mdc)
│     └─ NO  → Consider making it scoped — see below
└─ NO  → Does it apply to a specific file type or directory?
   ├─ YES → Can you express the scope as a glob pattern?
   │  ├─ YES → globs: src/**/*.ts + alwaysApply: false  (auto-attach)
   │  └─ NO  → Use a subdirectory .cursor/rules/ for that folder
   └─ NO  → Is it an opt-in standard or reference doc?
      ├─ YES → Is it used rarely / on specific tasks?
      │  ├─ YES → Manual — no globs, no alwaysApply  (use @rule-name)
      │  └─ NO  → Agent Decision — set description carefully
      └─ NO  → Does it encode workflow knowledge or scripts?
              └─ → It's probably a Skill, not a Rule (.cursor/skills/)
Best Practices & Anti-Patterns
✅ Best Practices
✦ One Rule, One PurposeSplit concerns into separate files: typescript.mdc, testing.mdc, security.mdc. Never put all conventions in one giant file — scoping stops working.
✦ Specific over Vague"All type aliases must use the Branded prefix" gets followed. "Write clean code" gets ignored. Vague rules are treated as suggestions by the model.
✦ Include Good + Bad ExamplesAdding a ✓ correct example and a ✗ incorrect example dramatically improves rule adherence vs. description alone.
✦ Use XML Tags for Long RulesGroup related instructions in <section> tags. LLMs follow structured XML-tagged content more reliably than flat markdown lists.
✦ Lint in CIAdd cursorrulespacks/cursor-lint-action to GitHub Actions. Broken frontmatter silently breaks rules — catch it before it ships to the whole team.
✦ Validate with npx cursor-lintRun locally before committing. Catches malformed YAML, missing frontmatter, wrong extensions — all silent failure modes.
❌ Anti-Patterns
⚠ Missing Closing ---The most common silent failure. YAML frontmatter must open AND close with ---. No error, no warning — rule simply doesn't load.
⚠ Wrong File Extensionrules.md or rules.txt will never be picked up. Only .mdc files in .cursor/rules/ are processed.
⚠ alwaysApply: true + GlobsWhen alwaysApply: true, globs are completely ignored. The rule loads for every file regardless. Don't expect glob scoping to work with always-on rules.
⚠ Monolithic Rule FilePutting all standards in one giant rules.mdc defeats glob scoping. TypeScript rules fire on Python files. Testing rules fire on config files.
⚠ Keeping .cursorrules alongside .mdcHaving both causes confusion. The .mdc wins silently. People editing .cursorrules wonder why nothing changes. Delete the legacy file.
⚠ Vague Descriptions for Agent-Decision ModeAgent-decision rules rely entirely on the description for activation. "General coding help" = never activates. "Apply when writing React components with hooks" = activates correctly.
