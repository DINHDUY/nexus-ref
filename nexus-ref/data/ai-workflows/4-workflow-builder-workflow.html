Module 1 — What does the Workflow Builder create? · 3 min
A meta-system that builds other agent systems
The Workflow Builder is not a trading agent or a support bot — it is the factory. You describe any professional workflow, and it outputs a complete, ready-to-run multi-agent system with research, a plan, agent files, and documentation.
The problem it solves
Building agents from scratch is slow
Researching a domain, designing subagents, writing YAML files, and creating docs typically takes 4–8 hours manually. The builder compresses this to a single orchestrator invocation.
The output it produces
A complete multi-agent system
Research markdown → plan markdown → .cursor/agents/*.md files → README.md. Everything another team needs to run, extend, and understand the system.
Works for any domain
Domain-agnostic by design
SSRN research, desktop support, pullback trading, RAG pipelines, HR onboarding — any structured professional workflow can be turned into agents using the same four-phase pattern.
◈
Key mental model: Think of the Workflow Builder as a compiler. You write a one-line description of a domain workflow (the "source code"), and it compiles that description into a running multi-agent system (the "executable").
Quick check — what is the output of a full Workflow Builder run?
AA single prompt template you can reuse in Claude
BA spreadsheet of workflow steps and owners
CResearch doc + plan + agent .md files + README, ready to run in Cursor
Module 2 — The 5-step human workflow · 3 min
How you drive the builder step-by-step
Each step has a specific AI mode (ASK / PLAN / AGENT) and a prompt template. Click any step to expand its prompt template, what the AI does, and the file it saves.
1
ASK / High-Think Model
Research the target domain workflow
Ask the AI to perform deep research on how professionals in a specific domain actually work — tools, steps, best practices, APIs.
Prompt template
Research a workflow that [who are the users] use [for what purpose]
Example 1 — SSRN research
Research a workflow that quant researchers use to search on SSRN for related papers
Example 2 — IT support
Research a workflow that desktop support engineers use to provide support for Windows applications
💾
Best practice: Save the research output to workflows/[project]/[name].md so you can review and manually edit before moving to Step 2.
2
PLAN / High-Think Model
Plan the subagents from the research
Give the AI the research markdown and ask it to map each workflow step to a specialized agent — including model choice, tools, I/O, and an orchestrator spec.
Prompt template
Plan for creating subagents for the workflow described in [path to research .md]
Example
Plan for creating subagents for the workflow described in workflows/lei/ssrn.md
◉
What the AI produces: Workflow-to-agent mapping table, pipeline ASCII diagram, per-agent specs (model, readonly, tools, I/O), and key design decisions.
3
AGENT / High-Think Model
Write the plan and build the agent files
In AGENT mode, the AI first saves the plan to a .md file, then creates each .cursor/agents/[name].md file with YAML frontmatter and full instructions.
Prompt template
First write the plan to [plan output path], and then build the agents
Example
First write the plan to workflows/ssrn/PLAN.md, and then build the agents
Agent file pattern (YAML frontmatter)
--- name: agent-name description: "one-sentence description for agent discovery" model: sonnet # or: fast readonly: false # true = read-only tool access --- ## Role Full instruction body with H2 sections for each major task...
4
AGENT / High-Think Model
Create the README.md documentation
Ask the AI to produce a concise, user-friendly README covering what the system does, the agent list, and practical invocation examples.
Prompt template
Add a README.md for the project, include what it is for and how to use it with some simple examples. Keep it simple and concise.
Exact prompt (use as-is)
Add a README.md for the project, include that it is for and how to use it with some simple examples. Keep it simple and concise.
◳
Why concise matters: The README is how future users discover what the agents do. A 2-page README gets read; a 20-page README gets skipped.
5
TEST / ITERATE
Test, test, and test the workflow
Run the orchestrator against a real use case. Identify where agents fail, where outputs are wrong, and refine prompts and agent instructions until objectives are satisfied.
⚡
Rule of thumb: Expect 3–5 refinement cycles before a new agent system performs reliably. Track failures in a simple list and fix the highest-frequency issue first.
Module 3 — The 5 automated agents · 3 min
When you run the orchestrator, these are the agents it uses
The Workflow Builder itself is a multi-agent system. The orchestrator coordinates four specialists that map directly onto the four manual steps you just learned — one agent per phase.
AgentModelPhaseRole
workflow.orchestrator sonnet All Single entry point. Accepts topic + project name + folder. Sequences all 4 phases, passes file paths between agents.
workflow.researcher sonnet Step 1 WebSearch + WebFetch to research the domain. Saves structured deep-dive markdown with steps, tools, APIs, and best practices.
workflow.planner sonnet Step 2 Reads research markdown. Maps workflow steps to agents. Designs orchestrator + subagent specs. Saves plan markdown.
workflow.builder sonnet Step 3 Reads plan markdown. Creates each .cursor/agents/[name].md file following the established YAML + instruction pattern.
workflow.documenter fast Step 4 Reads plan + agent list. Creates concise README.md: summary, agent list, invocation examples. Uses fast model — writing only, no reasoning required.
◈
File-based handoff is the key design decision. Agents share no conversation history. Each agent saves its output to a .md file; the next agent reads from that path. This is what makes the pipeline reliable and restartable at any phase.

Why does the workflow.documenter use the "fast" model while others use "sonnet"?
ACost savings — the documenter is optional so quality doesn't matter
BIts task is writing only — no domain reasoning required, so the fast model is sufficient and more efficient

Module 5 — Test & refine your system · 3 min
Building is 20% of the work. Testing is 80%.
Agent systems rarely work perfectly on the first run. The spec calls this out explicitly: "TEST TEST and TEST the workflow — refine until objectives are satisfied." Here is a practical testing protocol.
①
Run the simplest possible case first
Before testing edge cases, give the orchestrator a well-known, simple domain. Confirm each agent produces a valid file before trying a complex or novel workflow.
②
Review each phase output before proceeding
After the researcher saves its markdown, read it. Does it capture the actual workflow steps? Missing tools or steps here cascade into a poor plan and bad agents.
③
Check the YAML frontmatter in every agent file
The builder must match the exact YAML pattern of existing agents. Validate: name, description, model, readonly. Wrong or missing fields cause silent failures at runtime.
④
Test each subagent in isolation before running the orchestrator
Call each agent individually with controlled input. Confirm the output format before wiring them into the full pipeline. Isolation narrows the failure surface.
⑤
Log what fails and fix the highest-frequency issue first
Keep a running list of failures across test runs. One systemic issue (e.g. wrong file path format) often explains 60% of failures. Fix root causes, not symptoms.
Session hands-on checklist — build your first system
Choose a domain: write a one-sentence description of a workflow you know well
Run the researcher prompt using the template from Step 1
Review the research .md — edit any missing steps or tools before continuing
Run the planner and confirm the agent mapping makes sense
Build agents and validate YAML frontmatter in each file
Generate README and verify it is concise and accurate
Test one subagent in isolation — call it directly with sample input
Run the full orchestrator and log any failures for refinement
You now have the complete meta-workflow
Research → Plan → Build → Document → Test. Use the prompt templates from Module 4, validate each phase output before proceeding, and expect 3–5 refinement cycles. Every domain workflow can become an agent system — start with one you know deeply.
