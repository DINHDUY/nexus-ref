Overview & Quickstart
What Hooks Can Do
Observe
Audit every tool call, shell command, file edit, and MCP execution across the agent loop.
Control
Block risky operations, gate SQL writes, deny reads to sensitive files, stop prompt submissions.
Extend
Run formatters after edits, inject context at session start, auto-continue with follow-up messages.
Scan
Detect PII or secrets in files, scan AI-generated code for vulnerabilities before it lands.
INFO: CoverageWorks with both Cursor Agent (Cmd+K / Agent Chat) and Cursor Tab (inline completions). Each surface uses distinct hook event names.
Quickstart — Project Hook
FILE: <project>/.cursor/hooks.json 
{
  "version": 1,
  "hooks": {
    "afterFileEdit": [{ "command": ".cursor/hooks/format.sh" }]
  }
}
FILE: .cursor/hooks/format.sh 
#!/bin/bash
# Read input, do something, exit 0
cat > /dev/null
exit 0
WARNING: Working DirectoryProject hooks run from the project root: use .cursor/hooks/script.sh. User hooks (~/.cursor/) use ./hooks/script.sh.
TIP: Restart RequiredAfter editing hooks.json, restart Cursor. Check the Hooks tab in Settings for active hooks.
Hook Execution Types
Command-Based Hooks (default)

Execute shell scripts that receive JSON via stdin and return JSON via stdout.

hooks.json example with matcher 
{
  "hooks": {
    "beforeShellExecution": [{
      "command": ".cursor/hooks/approve-network.sh",
      "timeout": 30,
      "matcher": "curl|wget|nc"
    }]
  }
}
Exit CodeMeaning
0Success — use the JSON output from stdout
2Block the action (equivalent to deny permission)
otherHook failed — action proceeds (fail-open)
DANGER: Fail-Closed HooksbeforeMCPExecution and beforeReadFile are fail-closed: crash or timeout = action blocked. All other hooks are fail-open.
Prompt-Based Hooks (LLM-evaluated)

Use a fast LLM to evaluate a natural language condition. No script required.

hooks.json — prompt hook 
{
  "hooks": {
    "beforeShellExecution": [{
      "type": "prompt",
      "prompt": "Does this command look safe to execute?
Only allow read-only operations.",
      "timeout": 10
    }]
  }
}
FeatureDetail
Returns{ "ok": boolean, "reason": "..." }
$ARGUMENTSAuto-replaced with full hook input JSON
modelOptional field to override the LLM model
SpeedUses a fast model for low-latency evaluation
TIP: Use WhenPrompt hooks shine for policy enforcement where parsing would be complex — e.g. "Only allow read-only commands. Never delete or write."
Agent Hook Events
Session Lifecycle
EventWhen it fires
sessionStartNew composer conversation created. Can inject env vars and context, or block creation.
sessionEndConversation ends. Fire-and-forget — response is logged but not used.
sessionStart output fields 
{
  "env": { "KEY": "value" },    // optional env vars
  "additional_context": "...", // injected into sys ctx
  "continue": true,            // false = block session
  "user_message": "..."        // shown if blocked
}
Tool Use Hooks
EventWhen it fires
preToolUseBefore ANY tool. Can allow/deny and modify tool input. Use matchers to filter by tool type.
postToolUseAfter successful tool. Can modify MCP tool output seen by the model.
postToolUseFailureWhen tool fails, times out, or is denied. For error tracking and recovery.
preToolUse output 
{
  "decision": "allow" | "deny",
  "reason":   "shown to agent if denied",
  "updated_input": { "command": "npm ci" }
}
INFO: preToolUse MatchersFilter by tool type: Shell, Read, Write, Grep, Delete, MCP, Task
Shell and MCP Hooks
EventBehavior
beforeShellExecutionBefore any shell command. Returns permission: allow / deny / ask.
afterShellExecutionAfter shell completes. Gets full output and duration (ms).
beforeMCPExecutionBefore MCP tool. Fail-closed — crash = blocked.
afterMCPExecutionAfter MCP tool. Gets result JSON and duration (ms).
permission output schema 
{
  "permission": "allow" | "deny" | "ask",
  "user_message": "shown in UI",
  "agent_message": "sent to agent"
}
File Hooks
EventBehavior
beforeReadFileBefore agent reads a file. Fail-closed. Can deny with user message.
afterFileEditAfter agent edits a file. Gets path + array of {old_string, new_string} edits.
beforeReadFile output 
{
  "permission": "allow" | "deny",
  "user_message": "shown when denied"
}
WARNING: Access ControlUse beforeReadFile to block sensitive files (e.g. .env, private keys) from ever being sent to the model. Fail-closed for safety.
Subagent Hooks
EventBehavior
subagentStartBefore Task tool spawns a subagent. Can allow or deny creation.
subagentStopWhen subagent completes or errors. Can emit followup_message to loop.
subagentStart matcher — filter by type 
"matcher": "explore|shell"
// types: generalPurpose, explore, shell, browser-use
TIP: Loop PatternReturn followup_message from subagentStop to auto-submit the next user message. Max 5 auto follow-ups via loop_limit.
Prompt, Response and Stop Hooks
EventBehavior
beforeSubmitPromptAfter user hits send, before backend. Can block with user_message.
afterAgentResponseAfter agent completes an assistant message. Gets final text.
afterAgentThoughtAfter thinking block. Gets text + duration_ms. Observational.
preCompactBefore context compaction. Cannot block. Returns optional user_message.
stopAgent loop ends. Can emit followup_message for auto-retry or looping.
stop hook output — auto-loop 
{
  "followup_message": "Retry: check credentials."
  // loop_count in input = auto follow-ups fired (starts 0)
}
Tab Hooks and Full Event Reference
Tab-Specific Events

beforeTabFileRead

  • Fires only for Tab completions
  • Control file access before Tab reads
  • No attachments field
  • Returns permission: allow | deny
  • Apply stricter policy than Agent reads

afterTabFileEdit

  • Fires only for Tab completions
  • Run formatters or audit Tab code
  • Includes precise range per edit
  • Includes old_line and new_line
  • Finer granularity than afterFileEdit

Why Separate?

  • Tab operates autonomously
  • Different trust level than Agent
  • Separate audit trails per surface
  • Tab edits have line-level range data
  • Different redaction policies needed
afterTabFileEdit — extra fields vs afterFileEdit 
{
  "file_path": "/abs/path/file.ts",
  "edits": [{
    "old_string": "...", "new_string": "...",
    "range": { "start_line_number": 10, "end_line_number": 10 },
    "old_line": "line before edit", "new_line": "line after edit"
  }]
}

Input / Output Schemas
Base Input (All Hooks)

Every hook receives these fields in addition to hook-specific ones:

{
  "conversation_id": "string",  // stable across turns
  "generation_id":   "string",  // changes per user message
  "model":           "string",  // configured composer model
  "hook_event_name": "string",  // which hook is running
  "cursor_version":  "string",  // e.g. "1.7.2"
  "workspace_roots": ["path"], // project root(s)
  "user_email":      "string|null",
  "transcript_path": "string|null"
}
Env VariableDescription
CURSOR_PROJECT_DIRWorkspace root directory (always present)
CURSOR_VERSIONCursor version string (always present)
CURSOR_USER_EMAILAuthenticated user email (if logged in)
CURSOR_CODE_REMOTERemote-aware project path (remote workspaces)
CLAUDE_PROJECT_DIRAlias for project dir (Claude Code compat.)
TIP: Session Env VarsEnv vars set in sessionStart output are passed to all subsequent hook executions within that session.
Key Output Patterns
preToolUse — allow, deny, or modify input 
{
  "decision":      "allow" | "deny",
  "reason":        "shown to agent if denied",
  "updated_input": { "command": "npm ci" }
}
stop / subagentStop — auto-loop 
{
  "followup_message": "Retry: credentials checked."
  // loop_limit default: 5, null = unlimited
  // loop_count in input starts at 0
}
postToolUse — modify MCP output only 
{
  "updated_mcp_tool_output": { "redacted": true }
}
beforeSubmitPrompt — block the prompt 
{
  "continue":     false,
  "user_message": "PII detected — remove before sending."
}
Configuration and Matchers
Per-Script Options
OptionTypeDefaultDescription
commandstringrequiredShell command, absolute, or relative path
typestring"command""command" | "prompt"
timeoutnumberplatformExecution timeout in seconds
loop_limitnumber|null5Max auto follow-ups for stop/subagentStop hooks. null = no limit
matcherstring—Regex filter — matched field depends on hook type
INFO: Matcher Behavior by Hook preToolUse: matches tool type (Shell, Read, Write, Grep, Delete, MCP, Task)
subagentStart: matches subagent type (generalPurpose, explore, shell, browser-use)
beforeShellExecution: regex matched against the full command string

Practical Patterns
Audit All Events (Bash)
FILE: audit.sh — universal log hook 
#!/bin/bash
json_input=$(cat)
timestamp=$(date '+%Y-%m-%d %H:%M:%S')
mkdir -p "$(dirname /tmp/agent-audit.log)"
echo "[$timestamp] $json_input" >> /tmp/agent-audit.log
exit 0
TIP: Universal HookAttach to any event. Exit 0 = no-op audit. Zero logic, maximum observability. A great starting point for any hook.
Block Git, Allow gh (Bash)
FILE: block-git.sh — beforeShellExecution 
#!/bin/bash
input=$(cat)
command=$(echo "$input" | jq -r '.command // empty')
if [[ "$command" =~ git[[:space:]] ]]; then
  echo '{"permission":"deny",
    "user_message":"Use gh CLI instead.",
    "agent_message":"git blocked, use gh tool"}'
else
  echo '{"permission":"allow"}'
fi
WARNING: Requires jqInstall: brew install jq or apt install jq
Auto-Retry on Error (TypeScript / Bun)
FILE: track-stop.ts — stop hook 
// Track error count, emit retry followup
const payload = await parseHookInput();
entry.errorCount = payload.status === "error"
  ? entry.errorCount + 1 : 0;

if (entry.errorCount >= 2 && payload.loop_count < 4) {
  response.followup_message =
    "Retry triggered. Double-check credentials.";
}
wiring in hooks.json 
{ "stop": [{ "command": "bun run .cursor/hooks/track-stop.ts" }] }
INFO: Use TypeScript WhenYou need typed JSON, durable file I/O, HTTP telemetry, or conditional retry logic awkward in Bash.

Decision: Which Hook Event?
What do you want to intercept?
+- Shell commands       -> beforeShellExecution
+- MCP tool calls       -> beforeMCPExecution  (fail-closed)
+- File reads (agent)   -> beforeReadFile      (fail-closed)
+- Any tool (generic)   -> preToolUse + matcher
+- Prompt submissions   -> beforeSubmitPrompt
+- Subagent spawning    -> subagentStart
+- After file edit      -> afterFileEdit  (formatter)
+- Tab completions      -> beforeTabFileRead / afterTabFileEdit
+- Session setup        -> sessionStart
+- Auto-retry / loop    -> stop or subagentStop + followup_message
Troubleshooting
ProblemFix
Hook not firingRestart Cursor. Check the Hooks tab in Settings and the Hooks output channel.
Wrong working dirProject hooks: use .cursor/hooks/. User hooks: use ./hooks/.
MCP always blockedbeforeMCPExecution is fail-closed. Check script syntax, exit codes, and stderr.
Hook output ignoredNon-0/2 exit codes = fail-open (action proceeds). Fix the script crash.
Infinite loopSet loop_limit on stop/subagentStop hooks. Default max is 5 auto follow-ups.
Script not foundMake executable: chmod +x .cursor/hooks/script.sh (not needed on Windows)
INFO: Debug TipWrite debug logs to /tmp/hooks.log from your scripts. The Hooks output channel shows script errors and execution traces.
