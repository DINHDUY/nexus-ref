What Are Agent Skills?
📐 Skills vs Rules vs Commands

Skills are domain-specific procedural knowledge — they teach the agent how to perform a task. Unlike Rules (always-on constraints) or Commands (prompt templates), skills load on-demand when relevant.

Skills

  • SKILL.md + dirs
  • On-demand (lazy)
  • How-to knowledge
  • Auto + /slash menu
  • Has scripts/ dir
  • Open standard
  • 20+ agents

Rules

  • .cursor/rules/*.mdc
  • Always-on / matched
  • Style standards
  • Auto-applied
  • No scripts
  • Cursor only
  • Frontmatter driven

Commands

  • .cursor/commands/*.md
  • Manual invoke
  • Saved prompts
  • Type /name
  • No scripts
  • Cursor only
  • No frontmatter

⚡ Progressive Disclosure — 3-Tier Loading

Only what's needed gets loaded. Even with many skills installed, context windows stay lean.

Tier 1
Discovery — Metadata always loaded at startup
Agent reads name + description for all installed skills. Used to match skills to tasks automatically.
~100 tokensper skill, always
Tier 2
Activation — Full SKILL.md body loaded on match
When description matches the task, the full instructions body is loaded. Keep this under 500 lines / 5 000 tokens.
<5 000 tokensrecommended max
Tier 3
Resources — scripts/, references/, assets/ on demand
Files referenced in the body are loaded only when explicitly needed. Small focused files = less context used.
as neededdemand-loaded
✦ Dual activation Auto: agent keyword-matches description to your task. Manual: type /skill-name in Cursor v2.4+ slash menu, or say "use the X skill" explicitly.

SKILL.md — Complete Anatomy
📄 Full SKILL.md Template
📁 .cursor/skills/my-skill/SKILL.md 
─── YAML FRONTMATTER (required) ──────────────────
---
name: my-skill-name         # kebab-case, 1-64 chars
                              # MUST match parent folder name
description: >-              # 1-1024 chars  ← CRITICAL
  What this skill does AND when to use it.
  Include trigger keywords: e.g. "Use when
  working with PDF files, forms, or document
  extraction tasks."
license: MIT                 # optional
compatibility: >-            # optional: env requirements
  Requires git, docker, GITHUB_TOKEN env var.
  Designed for Cursor / Claude Code.
metadata:                    # optional: arbitrary KV
  author: your-org
  version: "1.2"
  category: devops
allowed-tools: Bash(git:*) Read Write  # optional, experimental
---
─── MARKDOWN BODY (instructions) ─────────────────

# My Skill

## When to Use
Detailed trigger scenarios (backs up description).

## Prerequisites
- Tools, packages, or env vars needed

## Steps
1. First action (be specific)
2. Second action
3. Verification step

## Rules & Constraints
- Hard rules the agent must follow

## Output Format
Exact format of expected output.

## References
See [detailed guide](references/REFERENCE.md)
Run: scripts/execute.sh
📋 Frontmatter Fields Reference
FieldReq?Constraints & Notes
name Required 1–64 chars. Lowercase + hyphens only. No leading/trailing hyphens. No --. Must match parent folder.
description Required 1–1024 chars. Describe WHAT + WHEN. Include trigger keywords. This is the routing logic — vague = never activates.
license Optional License identifier or path to bundled file. E.g. MIT, Apache-2.0, LICENSE.txt.
compatibility Optional 1–500 chars. Environment requirements, required tools, OS, network. Omit if no special requirements.
metadata Optional Arbitrary key-value map. Use for author, version, category, stack. Unique keys to avoid collisions.
allowed-tools Experimental Space-delimited pre-approved tools. Syntax: Bash(git:*) Read Write. Support varies per agent.
⚠ Name Validation — silent failures PDF-Processing → invalid (uppercase)  ·  -pdf → invalid (leading hyphen)  ·  pdf--proc → invalid (consecutive hyphens). Validate: skills-ref validate ./skill-name
✦ Description is the only routing key At discovery time, the agent reads only name + description for all skills. Include specific domain phrases users would naturally say: "migration", "deploy", "auth", "CSV", "refactor". The better the description, the more reliably the skill fires automatically.

── Method 1: Vercel Skills CLI (recommended) ───────
npx skills                       # interactive picker
npx skills add pdf-processing    # from registry by name
npx skills add github.com/org/repo/path
npx skills list                   # list installed
npx skills remove pdf-processing

── Method 2: agentskills.io registry ───────────────
npx @agentskills/cli install pdf-processing

── Method 3: Manual copy ────────────────────────── 
cp -r skill-folder/ .cursor/skills/

── Method 4: Git submodule (team shared) ───────────
git submodule add \
  https://github.com/org/skills.git \
  .cursor/skills/org-skills

── Method 5: Validate before using ─────────────── 
skills-ref validate .cursor/skills/my-skill

# ✓ Name matches directory
# ✓ Description non-empty

✦ Reliability tip Implicit auto-activation requires precise keywords in the description. For guaranteed activation, always use the explicit slash command or name the skill in your prompt.
Best Practices & Anti-Patterns
✅ Best Practices
✦ Description = Routing LogicWrite WHAT + WHEN. Include specific trigger keywords: "CSV", "migration", "deploy", "auth", "refactor". The agent's only signal during discovery.
✦ Keep SKILL.md under 500 linesMove long material to references/. Point to it explicitly. Agent loads those files only when needed, keeping context tight.
✦ Name Must Match FolderAlways validate with skills-ref validate ./skill-name before committing. Name mismatch silently breaks loading.
✦ Reference Files ExplicitlyWrite exact paths: "See references/edge-cases.md". Explicit references tell the agent the file exists and when to load it.
✦ Self-Contained ScriptsScripts should include error codes, helpful messages, and document all required env vars in the compatibility field.
✦ Use metadata for DiscoverabilityAdd author, version, category to metadata — makes skill libraries easier to audit and manage over time.
❌ Anti-Patterns
⚠ Vague Description"Helps with code." → never auto-activates. Be specific about tasks, domains, and keywords users would naturally use.
⚠ Invalid Name Formatname: PDF-Processing fails silently — uppercase not allowed. Only lowercase letters and hyphens. No leading/trailing/consecutive hyphens.
⚠ Name / Folder Mismatchname: code-review inside folder codeReview/ → silent failure. Name must exactly match folder name.
⚠ Embedding Everything in SKILL.mdA 2 000-line SKILL.md loads entirely into context on every activation. Move deep content to references/ — load it on demand.
⚠ Scripts Without Error HandlingScripts that fail silently leave the agent confused. Always include exit codes and descriptive error messages.
⚠ Confusing Skills with RulesAlways-on constraints (formatting, naming) belong in .cursor/rules/. Skills are for on-demand procedural know-how, not persistent standards.
