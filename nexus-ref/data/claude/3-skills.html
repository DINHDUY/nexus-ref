What Is a Skill — Mental Model
🧠 Core Mental Model

A skill is a SKILL.md file inside a named folder — it gives Claude a repeatable playbook for a specific task. You invoke it with a slash command, Claude invokes it automatically when relevant, or both. Think of it as a reusable prompt template with superpowers.

SKILL.md
instructions
→
/skill-name
slash invoke
→
Claude executes
uses its tools
→
Repeatable
same result
ℹ Skills = Commands (unified) .claude/commands/review.md and .claude/skills/review/SKILL.md both create /review and work identically. Skills are the new recommended system — commands still work but skills add: a supporting files directory, frontmatter to control invocation, and auto-activation by Claude. Both formats are maintained indefinitely.
✦ Start simple, iterate Your first skill can be 10 lines with no frontmatter at all. Put instructions you'd copy-paste manually into SKILL.md. Add frontmatter only when you need to control invocation or tool access.
🆚 Skills vs Other Extension Mechanisms
Skills
Repeatable workflows, slash commands, auto-activation

Invoke: /skill-name or auto
Location: .claude/skills/
Can use context: fork for subagent isolation
Subagents
Long-running agents with memory, specialized toolsets

Invoke: @agent-name or Task tool
Location: .claude/agents/
Persistent across the session
CLAUDE.md Rules
Always-on conventions and standards

Auto-loaded every session
Location: CLAUDE.md
No slash command — passive knowledge
🔑 When to use a skill vs CLAUDE.md Put ongoing conventions (code style, commit format) in CLAUDE.md. Put specific workflows (deploy, code review, PR summary) in a skill. Skills you invoke on demand; CLAUDE.md loads every session automatically.

Arguments & Dynamic Context Injection
📥 Arguments — $ARGUMENTS & Positional $0 $1 $2
Basic: $ARGUMENTS = everything after the slash command 
── Invoke: /code-review src/auth.ts ──────────────

---
name: code-review
description: Reviews a file for bugs and style
argument-hint: [file or PR number]
---

Review the file at $ARGUMENTS for:
- Security vulnerabilities
- TypeScript type safety
- Performance issues

# /code-review src/auth.ts → "$ARGUMENTS" = "src/auth.ts"
Positional: $0 $1 $2 for multi-word arguments 
── Invoke: /migrate-component SearchBar React Vue ─

---
name: migrate-component
description: Migrates a component from one framework to another
argument-hint: [component] [from-framework] [to-framework]
---

Migrate the component $0 from $1 to $2.
Preserve all existing behavior and tests.

# $0 = "SearchBar"  $1 = "React"  $2 = "Vue"
ℹ Auto-append if $ARGUMENTS missing If you don't include $ARGUMENTS in your skill body, Claude Code automatically appends it at the end as: ARGUMENTS: <value>. You don't have to include it in the SKILL.md if you only want it appended automatically.
⚡ Dynamic Context Injection — !`command` Syntax

The !`command` syntax runs a shell command when Claude loads the skill. The command's stdout replaces the backtick block. This lets you inject dynamic context that's always current when the skill runs.

Examples of backtick dynamic injection 
---
name: pr-summary
description: Summarizes the current pull request changes
context: fork
allowed-tools: Read Bash(git *)
---

## Current Context (auto-injected at load time)

- Branch: !`git branch --show-current`
- Diff:   !`git diff main --stat`
- Remote: !`git remote get-url origin`
- Issues: !`gh issue list --limit 5 2>/dev/null || echo "N/A"`

## Task
Write a PR summary for the changes above.
Include: what changed, why, testing done.
Reviewing: $ARGUMENTS
Environment variables available inside !` ` and skill body 
$ARGUMENTS             # all text after /skill-name
$0 $1 $2               # positional args (space-separated)
$CLAUDE_SESSION_ID     # current session identifier
$CLAUDE_PLUGIN_ROOT    # absolute path to skill's directory
                       # (plugin-bundled skills only)

── Use CLAUDE_PLUGIN_ROOT to reference scripts ───
Run the helper script:
${CLAUDE_PLUGIN_ROOT}/scripts/debug-helper.ts
⚠ Backtick commands run at load, not at invoke The shell command runs once when Claude loads the skill. It does not re-run on subsequent turns. For truly fresh data, use allowed-tools: Bash(git *) and let Claude run the commands in-session.

Full example: deploy skill with context: fork 
---
name: deploy
description: Deploy the application to production environment
context: fork
disable-model-invocation: true
allowed-tools: Read Write Bash(npm *) Bash(git *) Bash(docker *)
---

# Deploy Application
Target: $ARGUMENTS (defaults to staging if omitted)

## Steps
1. Run the test suite: `npm run test`
2. Build the application: `npm run build`
3. Check for uncommitted changes: `git status`
4. Tag the release: `git tag v$(date +%Y%m%d-%H%M)`
5. Push to deployment target
6. Verify health endpoint responds 200
⚠ context: fork only works for task skills with explicit steps If your skill contains guidelines like "use these API conventions" without an actionable task, the forked subagent receives the guidelines but no prompt to execute. It may return without meaningful output. Use context: inline for reference/knowledge content.
allowed-tools & Skill-Scoped Hooks
🔧 allowed-tools — Permission-Free Tool Access

Tools listed in allowed-tools run without permission prompts while the skill is active. Same syntax as the global permissions allow rules.

allowed-tools syntax examples 
── Read-only analysis ────────────────────────────
allowed-tools: Read Grep Glob

── File modifications ────────────────────────────
allowed-tools: Read Edit Write

── Git operations ────────────────────────────────
allowed-tools: Read Write Bash(git *)

── Testing workflows ─────────────────────────────
allowed-tools: Read Write Bash(bun test:*) Bash(npm test:*)

── Full development ──────────────────────────────
allowed-tools: Read Edit Write Bash(git *) Bash(bun *) Bash(npm *)

── MCP tools ─────────────────────────────────────
allowed-tools: Read mcp__github__create_pr mcp__puppeteer
✦ Scope tool access to the minimum needed Read Grep Glob for analysis skills. Add Edit Write only if the skill modifies files. Add Bash(git *) only for git workflows. Scoping tools makes skills safer and faster — Claude doesn't need to prompt for each tool use.
ℹ Tool access is scoped to skill activation The allowed-tools list only applies while this skill is being executed. It does not permanently change your session permissions. After the skill completes, normal permission rules resume.
🔗 Skill-Scoped Hooks in Frontmatter

Skills can define lifecycle hooks in their frontmatter. These hooks only run while this specific skill is active — they're automatically cleaned up when the skill finishes. once: true is supported here (unlike session-level hooks).

Skill hooks anatomy 
---
name: typescript-dev
description: TypeScript development with auto-lint
allowed-tools: Read Edit Write Bash(npm *)
hooks:
  PreToolUse:
    - matcher: "Bash"                  # regex pattern for tool name
      hooks:
        - type: command
          command: "./scripts/validate-cmd.sh"
          once: true               # run only once per session
  PostToolUse:
    - matcher: "Write(*.ts)|Edit(*.ts)"  # pipe = OR
      hooks:
        - type: command
          command: "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\""
        - type: command
          command: "npx eslint --fix \"$CLAUDE_TOOL_INPUT_FILE_PATH\" 2>/dev/null"
  Stop:                                 # auto-converts to SubagentStop for fork skills
    - hooks:
        - type: command
          command: "./scripts/completion-check.sh"
---
✦ once: true is skills-only once: true on a hook entry means it runs only once per session, even if the trigger fires many times. This field is supported in skill frontmatter hooks but is not available in session-level hooks in settings.json.
Auto-Activation — How Claude Decides to Load Skills
🤖 How Auto-Activation Works

At the start of each session, Claude loads the descriptions of all available skills into a discovery layer. When you send a message, Claude scans descriptions to decide which skill (if any) is relevant enough to load and activate.

Writing descriptions that trigger reliably 
── ✗ Too vague — rarely triggers ────────────────
description: Helps with files

── ✓ Specific with trigger keywords ─────────────
description: Parse and validate JSON files including schema
               validation. Use when working with JSON data,
               .json files, or configuration files.

── ✓ Action + domain + file type ─────────────────
description: Reviews Vue and React components against
               accessibility and performance criteria. Use when
               asked to review, audit, or improve a UI component.

── Description keyword triggers (add to description):
# File types:  .ts .json .md .pdf .csv .pptx
# Actions:     review, audit, parse, validate, deploy
# Domains:     API, database, auth, CI/CD, testing
# User words:  "how", "what", "explain", "fix", "create"
⚠ If skill isn't auto-activating 1. Check /context — is it in the budget? 2. Check description — is it specific enough? 3. Run /context to see "excluded-skills" warnings. 4. Add disable-model-invocation: true → check if it works when you invoke manually. 5. Description quality is almost always the cause.

✦ Keep descriptions short and precise Long descriptions consume more of the token budget. Aim for 1–3 sentences. Include exactly the trigger keywords users will say naturally — don't pad with explanation.
Bundled Skills — Built Into Every Claude Code Session
📦 Bundled (Built-in) Skills — Always Available

Bundled skills ship with Claude Code and require no installation. Unlike built-in commands (which execute fixed logic), bundled skills are prompt-based — they give Claude a detailed playbook and let it orchestrate the work using its tools. Any custom skill with the same name overrides a bundled skill.

Slash CommandWhat It DoesAuto-Activates?
/claude-api Loads Claude API reference for your project's language (Python, TypeScript, Java, Go, Ruby, C#, PHP, cURL). Covers tool use, streaming, batches, structured outputs, and common pitfalls. Also loads Agent SDK reference for Python and TypeScript. Yes — when your code imports anthropic, @anthropic-ai/sdk, or claude_agent_sdk
Document skills Built-in skills for creating and processing Word (.docx), Excel (.xlsx), PowerPoint (.pptx), and PDF files. Prompt-based orchestration using code execution. Yes — when asked to create/process those file types
ℹ Bundled vs Built-in Commands /help, /compact, /clear are built-in commands — they execute fixed hardcoded logic. Bundled skills like /claude-api are prompt-based — Claude reads a SKILL.md and orchestrates freely using its tools. Built-in commands cannot be overridden; bundled skills can.

Decision Tree — Designing Your Skill
🌿 Choose the Right Settings for Your Skill
What kind of content is this skill?
├─ Reference/knowledge (style guides, conventions, domain context)
│  └─ context: inline  (default — can see conversation history)
│     └─ Should it be hidden from the slash menu?
│        ├─ YES → user-invocable: false  (Claude auto-loads, you can't /invoke)
│        └─ NO  → leave defaults (both you and Claude can invoke)
└─ Task with explicit steps (deploy, test, generate, review)
   ├─ Does it have side effects (push, publish, send, deploy)?
   │  └─ YES → disable-model-invocation: true  (user-only control)
   │           └─ Needs clean isolated context?
   │              └─ YES → context: fork  (isolated subagent)
   │              └─ NO  → inline is fine (has conversation history)
   └─ No side effects (analysis, summarize, explain)?
      └─ Should Claude auto-activate it?
         ├─ YES → leave defaults, write a specific description with trigger keywords
         └─ NO  → disable-model-invocation: true

Does the skill need tools without permission prompts?
└─ YES → add allowed-tools: Read Grep Bash(npm *)  (minimum needed)

Does the skill need runtime dynamic context (git branch, env vars)?
└─ YES → use !`command` backtick injection in the skill body

Does the skill take user input after the /slash-command?
└─ YES → use $ARGUMENTS (all) or $0 $1 $2 (positional)
          → add argument-hint: [description] for autocomplete guidance
