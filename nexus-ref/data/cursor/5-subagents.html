01 Anatomy of a Subagent File
📄 Base Template
📁 .cursor/agents/my-agent.md  or  ~/.cursor/agents/my-agent.md
---  # YAML Frontmatter ─────────────────────────────────
name:         agent-name           # unique ID (optional, defaults to filename)
description:  "Expert in X. Use when [specific context]."
model:        inherit              # inherit | fast | sonnet | opus | haiku
readonly:     false                # true = no file writes
is_background: false              # true = runs non-blocking in bg
---  # System Prompt ──────────────────────────────────────

You are an [expert in X] specialized in [Y].

When invoked:
1. [First action to take]
2. [Second action]
3. [Third action]

Report [expected result format]:
- [Output format or metric]
⚙️ YAML Fields Reference
name
Unique ID. Used for explicit invocation. Defaults to filename if omitted.
description
CRITICAL. This is what the Agent reads to decide delegation. Be specific about when to use.
model
inherit (default) · fast · sonnet · opus · haiku · or full model ID string
readonly
false (default) – full access. true – restricts all file write operations.
is_background
false (default) – foreground, blocks main. true – non-blocking, state in ~/.cursor/subagents/
tools
(allowlist) Restrict to specific tools only. See tools section.
disallowedTools
(denylist) Block specific tools while allowing everything else.
✦ Tip Never add comments inside YAML frontmatter — a single # inline can silently break recognition.
02 Execution Modes
▶ Sequential (Pipeline)

One agent finishes → output feeds into next. Good for staged workflows.

🧠 Main Agent
↓ delegates task + context
⚙ planner-agent
↓ returns plan
⚙ coder-agent
↓ returns code
⚙ reviewer-agent
↓ returns verdict
✓ Final Result
# .cursor/agents/orchestrator.md
---
name: orchestrator
description: "Coordinates complex workflows
  across multiple specialists.
  Use for multi-phase projects."
---

You are a workflow orchestrator.

Sequence:
1. Use planner-agent to create a plan
2. Pass plan → coder-agent to implement
3. Pass code → reviewer-agent to validate
4. Compile and return consolidated output
⚡ Parallel (Fan-Out)

Main agent spawns multiple subagents simultaneously via the Task tool.

🧠 Main Agent (fan-out)
↙ ↓ ↘
⚙ tests-agent
⚙ docs-agent
⚙ lint-agent
↘ ↓ ↙
✓ Merge Results
# Main agent invokes all at once:
Run in parallel:
- tests-agent on /src/auth/
- docs-agent for AuthService class
- lint-agent on modified files

Merge all results and
report any blocking issues.
✦ Git Worktrees Use git worktrees for true parallel execution — multiple branches on filesystem simultaneously without merge conflicts.
🌙 Background (Non-blocking)

Runs independently while you continue working. State persisted to disk.

is_background: true
⚠ State Location Background subagents write state to ~/.cursor/subagents/ during execution. 
# .cursor/agents/bg-monitor.md
---
name: bg-monitor
description: "Monitors test suite and logs
  failures. Use immediately after
  code changes."
model: fast
is_background: true
readonly: true
---

You run tests silently and
report failures via a summary.
Never block the main thread.

Good for: test watchers, linters, log analyzers, doc generators.

03 Advanced Patterns
🤝 Hand-off Pattern

One specialized agent completes its task and hands control with structured context to the next. Clean role boundaries.

# .cursor/agents/handoff-planner.md
---
name: handoff-planner
description: "Plans feature implementation.
  Hands off to coder-agent when plan
  is ready. Use at project kickoff."
model: opus
readonly: true
---

You are a planning specialist.

Steps:
1. Analyze the codebase structure
2. Identify all files that need changes
3. Output a structured HANDOFF DOCUMENT:

## HANDOFF TO: coder-agent
### Files to modify: [list]
### Changes per file: [details]
### Acceptance criteria: [list]
### Risks: [list]

Signal coder-agent to begin.
✦ Key Principle The handoff document is the contract. Keep it structured and machine-readable so the receiving agent has zero ambiguity.
🔄 Reviewer / Verifier Loop

A dedicated verification agent checks output and either approves or triggers a retry cycle.

# .cursor/agents/verifier.md
---
name: verifier
description: "Validates completed work.
  Use after tasks are marked done
  to confirm implementations work."
model: sonnet
readonly: true
tools: [Read, Bash, Grep, Glob]
---

You are a verification specialist.

For each completed task:
1. Read the implementation files
2. Run tests: `npm test` or equiv.
3. Check edge cases manually
4. Output one of:
   ✓ APPROVED — [reason]
   ✗ REJECTED — [specific issue]
     RETRY-INSTRUCTION: [fix needed]
main agent → coder-agent → verifier → if REJECTED → coder-agent
🌳 Hierarchical (Nested) Agents

A subagent itself acts as an orchestrator and spawns its own sub-subagents via CLI.

# Tier 1: Main orchestrator
name: project-manager

  # Tier 2: Domain coordinator
  name: backend-lead
    # Tier 3: Specialists
    name: db-schema-agent
    name: api-designer
    name: auth-specialist

  # Tier 2: Domain coordinator
  name: frontend-lead
    # Tier 3: Specialists
    name: component-builder
    name: stylist-agent
⚠ Performance Warning Performance degrades beyond 8 concurrent subagents. Keep hierarchy shallow — max 2–3 tiers.
💾 Resumable (Stateful) Agent

Long-running agents can be paused and resumed using their agent ID.

# CLI: spawn and capture agent ID
cursor-agent "run migration analysis" \
  --agent migration-analyst \
  --output-id

# Returns: agent_id=abc123

# Resume later with preserved context
cursor-agent "continue where you left off" \
  --resume abc123
# .cursor/agents/migration-analyst.md
---
name: migration-analyst
description: "Analyzes DB migrations.
  Long-running. Resumes from last
  checkpoint if given an agent ID."
model: sonnet
is_background: true
---

migration file. Write status to
~/.cursor/subagents/migration-state.json

05 Tool Permissions & Scoping
🔧 Tool Allowlist vs Denylist
# Allowlist — ONLY these tools:
---
tools: [Read, Grep, Glob, Bash]
---

# Denylist — everything EXCEPT these:
---
disallowedTools: [Write, Edit, Delete]
---

# MCP Server restriction:
---
tools: [Read, mcp:playwright, mcp:github]
---

File reads, search
Write/Edit
File modifications
Bash
Terminal commands
Grep/Glob
Pattern search
mcp:*

06 Invocation Patterns
💬 In-Chat Invocation
# Explicit (always works — recommended)
"Use the security-auditor subagent on the payment module"
"Ask the doc-writer to generate API docs"
"Have verifier check the auth flow"

# Implicit (description-driven auto-delegation)
The main agent reads all descriptions
and delegates automatically when:
  description: "Use immediately after writing code"
  description: "Use proactively when tests are failing"

# Chaining in conversation
"Implement login feature then have verifier check it,
and if APPROVED run doc-writer on AuthService"
⚡ Pro Tip Implicit auto-delegation is unreliable — explicitly name the subagent in your prompt for guaranteed invocation.
⌨️ CLI Invocation (Headless)
# Install Cursor CLI
curl https://cursor.com/install -fsS | bash

# Basic subagent spawn
cursor-agent "run full test suite analysis" \
  --agent test-runner

# Headless parallel execution
cursor-agent "refactor auth module" \
  --agent refactor-worker \
  --headless

# Spawn subagent + capture ID for resume
cursor-agent "analyze migrations" \
  --agent migration-analyst \
  --output-id > agent_id.txt

# Resume with preserved context
cursor-agent "continue analysis" \
  --resume $(cat agent_id.txt)

07 Decision Tree — Which Pattern to Use?
🌿 Pattern Selection Guide
Is the task complex with multiple steps?
├─ YES → Does it require isolated context / specialized knowledge?
│  ├─ YES → Are subtasks independent of each other?
│  │  ├─ YES → PARALLEL pattern  (fan-out, simultaneous execution)
│  │  └─ NO  → Does output of A feed into B?
│  │     ├─ YES → SEQUENTIAL / HAND-OFF pattern  (pipeline)
│  │     └─ NO  → HIERARCHICAL pattern  (orchestrator spawns sub-orchestrators)
│  └─ NO  → Is it a repeatable one-purpose action?
│     ├─ YES → Use a SKILL (.cursor/skills/) not a subagent
│     └─ NO  → Single subagent  with clear scoped prompt
└─ NO  → Is it a single, one-off action?
   ├─ YES → Is it repeatable/reusable?
   │  ├─ YES → Use a SKILL or slash command
   │  └─ NO  → Handle inline in main agent
   └─ NO  → Is it long-running / non-blocking?
      ├─ YES → BACKGROUND subagent  (is_background: true)
      └─ NO  → RESUMABLE subagent  (with agent ID for continuation)
08 Best Practices & Anti-Patterns
✅ Best Practices
✦ Specific Descriptions "Use when implementing OAuth flows with external providers" beats "Use for auth tasks". The description is the routing logic.
✦ Single Responsibility Each subagent does ONE thing well. Combine with patterns for complex tasks.
✦ Explicit Invocation Always name the agent explicitly in prompts. Implicit auto-delegation is unreliable.
✦ Start Small Begin with 2–3 focused subagents. Add more only when there's a genuinely distinct use case.
✦ Model Right-sizing Use fast/haiku for simple execution agents, opus for planning/reasoning agents.
✦ Git Worktrees for Parallel Enable true filesystem parallelism without conflicts when multiple subagents touch the same repo.
❌ Anti-Patterns
⚠ Vague Descriptions "Use for general tasks" → agent never gets auto-delegated. Be hyper-specific.
⚠ YAML Comments Never use # inline comments inside the frontmatter block — silent breakage guaranteed.
⚠ Overly Long Prompts A 2000-word system prompt is harder to maintain and slower. Target 200–400 words.
⚠ Too Many Subagents Performance degrades past 8 concurrent. Don't create an agent for everything.
⚠ Duplicating Slash Commands If it's a single-purpose action without context isolation → use a skill, not a subagent.
⚠ Mixed Responsibilities "Backend + security + docs agent" → split into 3 agents. One agent, one domain.

🔄 Agent Lifecycle
Load
Session start or /agents command
Invoke
Explicit name in prompt or auto-delegation
Execute
Own context window, own thread
Return
Result value back to main agent only
Reject
If main agent rejects → result is discarded, gone
Resume
Background only, via agent ID
⚠ Context Isolation Each subagent starts with a clean slate. Context from the main conversation is not automatically available — pass it explicitly.
