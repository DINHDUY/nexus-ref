How Commands Work
📁 What are Custom Commands?

Custom commands are plain Markdown files stored in .cursor/commands/. They are reusable AI prompts — like keyboard shortcuts for complex, multi-step tasks. Type / in Agent chat to browse and run them instantly.

Commands

  • Plain Markdown
  • Manually invoked
  • Reusable prompts
  • Team-shareable
  • No frontmatter

Rules

  • .mdc with frontmatter
  • Auto-applied / always-on
  • Persistent context
  • Pattern-triggered
  • In .cursor/rules/

Skills

  • SKILL.md files
  • Dynamic discovery
  • Procedural how-to
  • Slash menu invoke
  • In .cursor/skills/

📂 File Locations & Discovery
# PROJECT commands (version-controlled)
your-project/
└── .cursor/
    ├── commands/         ← custom slash cmds
    │   ├── code-review.md
    │   ├── create-pr.md
    │   └── run-tests.md
    ├── rules/            ← rules (.mdc files)
    ├── skills/           ← skills (SKILL.md)
    └── agents/           ← subagents

# GLOBAL commands (personal machine)
~/
└── .cursor/
    └── commands/         ← user-global cmds
        ├── my-workflow.md
        └── daily-standup.md
✦ Auto-merge Cursor scans both directories and merges results. Project commands take priority over global ones with the same filename.
⚠ No Frontmatter Unlike rules and subagents, .cursor/commands/ files use plain Markdown only — no YAML frontmatter. Adding frontmatter causes silent failures.
Anatomy of a Command File
📄 Base Template Structure
📁 .cursor/commands/my-command.md 
# [Command Title]
                        ↑ becomes the menu display name

## Objective
Brief description of what this command does
and what problem it solves.

## Context
What context the AI should be aware of:
- Current tech stack / framework conventions
- Project architecture expectations
- Coding standards to enforce

## Requirements
- Specific constraint or rule #1
- Specific constraint or rule #2
- Expected output format

## Steps
1. First thing the AI should do
2. Second step
3. Final action / report format

## Output Format
Describe the exact format of the response:
- e.g. structured markdown with ## headers
- e.g. inline code suggestions
- e.g. JSON output with schema X

## Examples (optional)
Show an example of expected output here.
✍️ Writing Effective Commands
PrincipleWhy It Matters
Single objectiveOne command = one task. Split multi-purpose into separate files.
Describe output formatAI uses output format as the success criterion. Be specific.
Include examplesAn example of expected output reduces hallucination dramatically.
State assumptionsName the framework, language, test runner, linter being used.
Kebab-case filenameFilename becomes the slash command name. Use create-pr.md not Create PR.md.
Active voice, verbs"Analyze", "Generate", "Fix" — not "This command is for..."
Version control itTreat command changes like code changes. Review in PRs.
✦ Context is automatic Commands always have access to the open files, selected code, and indexed codebase. You don't need to re-specify the project layout in every command.
Built-in System Slash Commands

These are Cursor's own built-in commands — not stored in files, always available in any chat.

⚡ Built-in Commands (Chat / Agent / CLI)
/summarize Compress long conversation to free up context window. Use when nearing token limit without wanting to start a new chat.
/models List all available AI models and switch between them mid-conversation. Available in CLI (Jan 2026) and editor.
/rules Create a new rule or edit existing rules directly from CLI without opening a file. CLI only (Jan 2026).
/mcp enable Enable a specific MCP server on the fly mid-session. CLI only. E.g. /mcp enable playwright
/mcp disable Disable an active MCP server mid-session without restarting. CLI only.
/[custom-name] Any .cursor/commands/name.md file becomes /name. Type / to see all available custom commands in a dropdown.
/[skill-name] Skills (SKILL.md files, v2.4+) appear in the slash command menu and can be explicitly invoked this way.
/agents Opens the subagent management panel to view, enable, and configure available subagents.
ℹ Source /summarize was introduced in v1.6 (Sep 2025). /models, /rules, /mcp enable|disable were added in the CLI Jan 8, 2026 release. Skills in slash menu added in v2.4 (Jan 2026).
CLI Commands (Terminal / Headless)
⌨️ cursor-agent CLI Reference
# Install Cursor CLI
curl https://cursor.com/install -fsS | bash

━━━━ BASIC USAGE ━━━━━━━━━━━━━━━━━━━━━━━━

# Run a task with agent
cursor-agent "your prompt here"

# Use a specific subagent
cursor-agent "fix all TypeScript errors" \
  --agent coder-agent

# Headless (background) mode
cursor-agent "run full test suite" \
  --headless

# List all available models
cursor-agent models
cursor-agent --list-models

# Use a specific model
cursor-agent "refactor this module" \
  --model claude-sonnet-4

# Resume a stateful session
cursor-agent "continue migration analysis" \
  --resume <agent-id>

# Output agent ID for resuming later
cursor-agent "analyze migrations" \
  --output-id | tee agent.id

━━━━ HOOKS (background scripts) ━━━━━━━━

# Hooks run after every file edit (10-20x
# faster since Jan 2026 — parallel exec)

# .cursor/hooks/on-edit.sh
#!/bin/bash
eslint $CURSOR_EDITED_FILE --fix
git add $CURSOR_EDITED_FILE
✦ Hooks vs Commands Hooks run automatically after every edit (event-driven). Commands are manually triggered. Use Hooks for enforcement, Commands for explicit workflows.

🔍 Code Review
QualityManual 📁 .cursor/commands/code-review.md 
# Code Review

## Objective
Perform a thorough code review of the
currently open or selected files.

## Review Checklist
Evaluate these areas in order:

**Logic & Correctness**
- Edge cases and null handling
- Off-by-one errors, boundary checks
- Race conditions or async pitfalls

**Code Quality**
- DRY violations or unnecessary coupling
- Naming clarity and consistency
- Function length and single-responsibility

**Performance**
- Unnecessary re-renders or re-computations
- Missing memoization or caching
- N+1 query patterns

**Security**
- Input validation and sanitization
- Auth checks on sensitive operations
- Secrets or sensitive data in code

## Output Format
For each issue found:
- **[SEVERITY]** — BLOCKER / WARNING / NIT
- File and line number
- Issue description
- Suggested fix with code snippet

End with: Overall verdict APPROVE or
REQUEST_CHANGES with a 2-line summary.

🧪 Run Tests & Fix
TestingAutomated 📁 .cursor/commands/run-tests-and-fix.md 
# Run All Tests and Fix

## Objective
Execute the full test suite, diagnose
failures, fix them, and re-run until
all tests pass.

## Steps
1. Run the test suite:
   - npm test / pytest / go test / etc.
   - Detect the runner from package.json
     or project config files

2. For each failing test:
   a. Read the error output carefully
   b. Identify root cause (not symptoms)
   c. Make the targeted fix
   d. Do NOT change test assertions unless
      the test itself is clearly wrong

3. After all fixes, re-run the full suite
   to confirm zero regressions

4. Report: X tests fixed, Y skipped,
   total pass rate

## Rules
- Never delete tests to make them pass
- One focused fix per failure
- Add a comment if fix is non-obvious
- Stop after 3 iterations if blocked
🧹 Lint & Format Fix
QualityQuick 📁 .cursor/commands/lint-fix.md 
# Lint Fix

## Objective
Run the linter and auto-formatter on the
current file or changed files, then fix
any remaining issues that can't be
auto-fixed.

## Steps
1. Detect lint tools from project config:
   - ESLint: .eslintrc* or eslint.config.*
   - Prettier: .prettierrc*
   - Biome: biome.json
   - Ruff: ruff.toml / pyproject.toml

2. Run auto-fix:
   npx eslint --fix [file]
   npx prettier --write [file]

3. For remaining errors that need manual
   fix, read the rule docs and apply the
   correct code pattern

4. Verify: re-run linter, confirm 0 errors

## Output
List auto-fixed issues and manually-fixed
issues separately. Flag any that need
human review.

# Security Audit

## Objective
Perform a targeted security review of
the current code changes or selected
files for common vulnerability patterns.

## Checklist
☐ SQL injection — check all DB queries
  for string concatenation / lack of
  parameterization
☐ XSS — check template rendering for
  unescaped user input
☐ CSRF — verify state-changing endpoints
  have CSRF tokens
☐ Auth — ensure all sensitive routes
  have authentication middleware
☐ Secrets — scan for hardcoded keys,
  tokens, passwords
☐ Dependency CVEs — run:
  npm audit / pip check / bundle-audit
☐ Input validation — all user inputs
  must be validated before processing
☐ Path traversal — check file operations

## Report Format
## Security Findings
| Severity | Issue | File:Line | Fix |
|----------|-------|-----------|-----|
Severity: CRITICAL / HIGH / MEDIUM / LOW
End with: PASS or FAIL + top 3 actions
📖 Generate API Docs
DocsRead-only 📁 .cursor/commands/generate-api-docs.md 
# Generate API Documentation

## Objective
Create comprehensive API documentation
for the current codebase or selected
API files.

## Steps
1. Identify all public API endpoints,
   functions, or exported members
2. For each endpoint / function extract:
   - Method + path (for REST APIs)
   - Parameters with types + descriptions
   - Return type + shape
   - Error codes and when they occur
   - Side effects
3. Generate concrete usage examples
4. Write a short description that a
   junior dev can understand

## Output Format
Use OpenAPI 3.0 format for REST APIs.
Use JSDoc / TSDoc for TypeScript.
Use docstring format for Python.
Place inline in source files OR output
to /docs/api.md — detect from project.

## Requirements
- Include auth requirements
- Include rate limiting info if present
- Include at least one real example
🐛 Debug Issue
DebugManual 📁 .cursor/commands/debug-issue.md 
# Debug Issue

## Objective
Systematically isolate and fix the bug
described by the user or visible in the
current error output.

## Debugging Process
1. READ the full error message and stack
   trace if available
2. Identify the FAILING ASSERTION vs the
   ROOT CAUSE — fix the root, not the
   surface symptom
3. Trace the code path:
   - Follow the call stack backwards
   - Check every data transformation
   - Verify assumptions about input types
4. Form a hypothesis and test it with
   minimal targeted code change
5. Verify the fix doesn't break adjacent
   behavior

## Do Not
- Do not add console.log and forget to
  remove them
- Do not suppress errors with try/catch
- Do not change unrelated code

## Output
Root cause: [explanation]
Fix applied: [description]
Files changed: [list]
Regression risk: LOW / MEDIUM / HIGH
♻️ Refactor Code
RefactorStaged 📁 .cursor/commands/refactor-code.md 
# Refactor Code

## Objective
Improve the structure, readability, and
maintainability of the selected code
without changing external behavior.

## Refactoring Targets
Apply these in order of impact:

1. **Extract functions** — any block > 20
   lines that does one thing
2. **Rename for clarity** — variables and
   functions that require mental decoding
3. **Remove duplication** — apply DRY to
   repeated patterns across files
4. **Simplify conditionals** — flatten
   nested ifs, use early returns
5. **Dependency direction** — ensure
   lower-level modules don't import
   higher-level ones
6. **Type safety** — add missing types
   or remove `any` in TypeScript

## Hard Rules
- Run tests after each change
- One refactor type per commit
- Do not change public API signatures
- Do not add new features in this pass

🗄 Database Migration
DBCareful 📁 .cursor/commands/database-migration.md 
# Database Migration

## Objective
Plan, create, and validate a database
schema migration based on the requested
change, including a rollback path.

## Steps
1. Read existing schema files to
   understand current structure
2. Identify all affected tables and
   foreign key relationships
3. Determine if migration is:
   - Additive (safe) — add column/table
   - Destructive (dangerous) — drop/alter
   - Data migration (risky) — transforms

4. Generate migration file with:
   - UP migration (apply)
   - DOWN migration (rollback)
   - Idempotency check where possible

5. Write validation query to confirm
   migration ran correctly

## Safety Rules
- NEVER drop columns without 2-phase plan
- Always include indexes for new FKs
- Test rollback before applying to prod
💬 Address PR Comments
GitReview 📁 .cursor/commands/address-pr-comments.md 
# Address GitHub PR Comments

## Objective
Process all open review comments on the
current PR and implement the requested
changes or craft a thoughtful response.

## Steps
1. Read all open PR comments
   (user provides or fetches via MCP)
2. For each comment:
   a. Understand what the reviewer wants
   b. Decide: IMPLEMENT or DISCUSS
   c. If IMPLEMENT: make the change
   d. If DISCUSS: write a clear response
      explaining the tradeoff or asking
      for clarification

3. After all changes, create a summary
   comment like "Addressed review by..."

## Response Tone
Professional, collaborative, and brief.
Never defensive. Never dismissive.
When disagreeing, explain the tradeoff
and offer to discuss further.

## Output
List of: Comment → Action taken / Reply

# SKILL.md format (different from commands)
# Skills USE frontmatter:
---

description: "Deploy to preview env.
  Use before sharing with stakeholders."
---

# Procedural how-to content follows...
# Skills = HOW-TO knowledge
# Commands = PROMPT templates
ℹ Key Difference Commands are static prompt templates. Skills teach the agent how to perform a task — they include environment-specific knowledge, scripts, and conditional instructions that vary per project.

Best Practices & Anti-Patterns
✅ Best Practices
✦ One Command, One GoalCreate code-review.md + security-review.md separately rather than one giant "review everything" command. Focused commands produce better results.
✦ Descriptive FilenamesThe filename becomes the slash command name. fix-compile-errors.md is better than fix.md. Names should be self-documenting.
✦ Include Output FormatAlways specify what the response should look like (table, markdown headers, code blocks, JSON). This is the most impactful single thing you can add.
✦ Ship with the RepoCommit .cursor/commands/ to git. New team members get the full command library instantly on clone.
✦ /summarize EarlyDon't wait for Cursor to auto-summarize. Use /summarize proactively before starting a new major sub-task within a long session.
✦ Hooks for Enforcement, Commands for IntentUse Hooks to automatically run linting after every save. Use Commands when you want to trigger a deliberate, multi-step workflow.
❌ Anti-Patterns
⚠ No Frontmatter in CommandsCommands are plain Markdown. Adding YAML frontmatter (--- blocks) will silently break the command or cause unexpected behavior.
⚠ Generic Command Nameshelp.md, fix.md, do-stuff.md — unrecognizable names mean the command never gets used. Always name by specific purpose.
⚠ Mixing Rules and CommandsDon't put always-on constraints in commands (they won't always run). Always-on conventions belong in .cursor/rules/, not commands.
⚠ Vague Output FormatA command that doesn't specify an output format will produce inconsistent results. Always include an "## Output Format" section.
⚠ Commands vs Skills ConfusionDon't put procedural "how the CI works" knowledge in commands — that belongs in Skills. Commands are prompt templates, not documentation.
⚠ No ExamplesFailing to include an example of expected output is the #2 cause of poor command results (after vague objectives). Even a 3-line example helps enormously.
